# 기출문제 연습프로그램 설계 문서

## 1. 시스템 개요
- 목표: PDF 기출문제와 정답 텍스트를 입력받아 오프라인에서 문제 연습을 지원하는 단일 사용자 MAUI 앱을 제공한다.
- 지원 플랫폼: 데스크톱 및 모바일
- 저장 방식: 로컬 저장소(JSON 파일 기반, 추후 SQLite 전환 검토)

## 2. 기능 요구사항
### 2.1 필수 기능
- 카테고리 단위 문제 묶음 관리
- PDF 업로드(이미지 기반 분석) 및 문제 분리(현재는 OCR 분석까지, 문제 분할 규칙은 미정)
- 정답 파일 업로드 및 매핑
  - 정답 형식: `문항번호:정답` 또는 `문항번호:정답1|정답2`
  - 객관식은 숫자 정답을 기반으로 채점
- 연습 실행
  - 카테고리 선택 후 출제 개수 지정
  - 최대 출제 개수는 카테고리 등록 문항 수로 제한 표시
  - 랜덤 출제
- 즉시 채점
  - 객관식: 사용자 입력은 숫자 한 개, 정답 후보 집합 중 하나와 일치 시 정답
- 주관식: 입력값 `Trim()` 후 대소문자 정규화(영문 Upper)
- 결과 표시
  - 총 문제 수 대비 정답 개수
- 오답 문제 모음
  - 오답만 조회
  - 재도전 시 정답이면 오답 목록에서 삭제
  - 사용자 수동 삭제 가능

### 2.2 제약/정책
- OCR 품질이 매우 낮으면 해당 PDF는 처리 불가로 표시
- OCR 추출은 플랫폼 공통 파이프라인으로 처리하도록 구성. 현재는 Windows와 외부 CLI 연동(예: pdftoppm + tesseract) 기반 경로를 우선 적용하고, 모바일 플랫폼도 동일 인터페이스로 처리하되 의존성 설치/설치 정책은 배포 시점에 맞춰 확정한다.
  - iOS/macOS는 PDFKit 텍스트 추출 기반 경로를 우선 적용(텍스트 기반 PDF). 이미지형 PDF의 경우 추후 Vision/MLKit 바인딩 기반 OCR로 확장 예정
  - Android/macOS는 `pdftoppm` 및 `tesseract` 배포/패키징 정책을 맞춰야 분석 가능
- 동일 파일명 업로드 시 덮어쓰기 동의 확인 후 진행
- 오프라인 필수 동작
- 단일 사용자

### 2.3 광고 기능(선택)
- 대상 플랫폼: Android / iOS 모바일 앱 모두
- 목적: 앱 실행 화면에 광고(예: 배너/전면광고)를 노출할 수 있는 구조를 확보
- 우선 채택 방식: MAUI용 AdMob 플러그인(`Plugin.MauiMTAdmob`) 기반으로 우선 구현
- 운영 모드에서는 실 광고 단위 ID 사용, 개발·테스트 단계는 테스트 광고 ID 사용
- 사용자 동의가 필요한 지역(예: EEA/US ATT 정책 적용 국가)은 동의 플로우를 별도 처리
- 광고 실패 시에도 핵심 문제풀이 기능은 정상 동작해야 함(광고 의존 제거)

### 2.4 광고 연동 설계(안)
- 구성 원칙
  - Android/iOS 각각 앱 ID와 광고 단위 ID를 분리 관리
  - 앱 시작 시 광고 SDK 초기화, 연습 화면에서 배너 노출, 필요 시 전면광고 노출
  - 실패/미지원 기기에서는 광고가 없어도 앱 동작 유지

- 플랫폼별 반영 파일(단일 프로젝트 구조 기준)
  - Android: `Platforms/Android/AndroidManifest.xml`
    - `com.google.android.gms.ads.APPLICATION_ID` 메타데이터 등록
    - 필요 시 AdMob `AdActivity` 관련 manifest 속성 검토
  - iOS: `Platforms/iOS/Info.plist`
    - `GADApplicationIdentifier` 등록
    - `SKAdNetworkItems` 및 광고 관련 식별자 검토/등록

- MAUI 초기화 흐름(요약)
  - `MauiProgram`에서 플러그인 초기화 호출
  - 광고 화면(예: 연습 화면 하단)에 배너 뷰 배치
  - 광고 단위별 노출 주기 제한(연타 클릭/중복 노출 방지) 정책을 명시적으로 적용

- 검증 및 배포 고려사항
  - Android: 실제 기기/에뮬레이터에서 테스트 광고가 정상 노출되는지 확인
  - iOS: Xcode 기반 빌드 환경에서 `Info.plist` 반영 상태와 광고 초기화 동작을 확인
  - 정책/심사: 스토어 정책 및 AdMob 가이드라인(개인정보/동의/트래킹) 준수

- 미확정/확실하지 않음
  - MAUI 공식 광고 통합 방식은 버전별/플러그인별로 상이할 수 있어 대상 SDK 버전 호환성은 적용 직전에 재확인 필요

## 3. 정답 규칙
- `MultipleChoice`(객관식): 문자열을 `:` 기준으로 정답을 파싱하여 숫자 배열로 저장
- 정답이 여러 개인 경우 (`|` 구분) 사용자 정답과 교집합이 있으면 정답 처리
- `Subjective`(주관식): 양끝 공백만 제거한 뒤 `ToUpperInvariant` 비교
- `문항 파일명`은 수동 입력이 아닌 **앱 로컬 `QuestionSourceFiles` 디렉토리에서 읽은 PDF 파일 목록**에서 선택되며,
  정답 맵 반영 시 해당 파일명을 `SourceFileName`으로 사용한다.

## 4. 핵심 데이터 모델(안)
- `Category`
  - `Id`, `Name`
- `Question`
  - `Id`, `Index`, `CategoryId`, `SourceFileName`, `Type`, `Prompt`, `Choices`, `CorrectAnswers`
- `PracticeQuestion`
  - `CategoryId`, `Question`, `UserAnswer`, `IsCorrect`

## 5. 계층 구조(예정)
- Presentation
  - 페이지: 시작, 카테고리, 업로드, 연습, 오답노트
  - ViewModel: 상태 관리, 커맨드, 화면 바인딩
- Domain
  - 채점 규칙, 정답 비교 정책, 랜덤 출제 규칙
- Infrastructure
  - 정답맵 파서, OCR 어댑터, 로컬 저장소 레포지토리(JSON)
- Application
  - 워크플로우 오케스트레이션

## 6. 주요 흐름
1. PDF 업로드
   - 사용자가 PDF 업로드 -> OCR 분석 실행 -> OCR 페이지 텍스트 저장(문제 분할은 별도 단계)
2. 정답 매핑
   - 정답 맵 텍스트 파싱 -> `(문항번호, 정답 목록)` 생성
   - 문제/정답 매핑 후 로컬 저장
3. 연습 시작
   - 카테고리에서 문항 선택 수 입력 -> 랜덤으로 문제 추출
   - 입력 받자마자 즉시 채점하여 정오답 표시
4. 종료
   - 정답 수 집계 후 화면 표시
5. 오답 노트
   - 정답 실패 항목 저장
   - 재학습 시 정답이면 삭제

## 7. 개발 우선순위
- 1단계: MVVM 구조 완성 및 카테고리/문항 도메인 모델 정리
- 2단계: 정답 비교 서비스 고도화(정규화/여러 정답)
- 3단계: 로컬 저장소 안정화 및 오답 노트 고도화
- 4단계: PDF OCR 파싱 통합
- 5단계: UI 품질 개선 및 사용성 보완

## 8. 미확정/확실하지 않은 항목
- PDF 문제 분리 규칙은 기출별로 형식이 달라, 자동 분할 정확도는 확정 불가
- 스캔본/이미지 잡음이 큰 경우 실패 판정 기준(예: 최소 신뢰도 임계값)은 추후 실데이터 기반으로 조정 필요
